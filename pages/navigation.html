User Menu

Composant standalone pour afficher l'identité utilisateur dans le header des apps M3. Trigger avatar + caret, dropdown avec info session, lien "Mon compte" (Authentik) et déconnexion POST. Configurable via attributs data-* ou via initUserMenu(el, options).

Initials — 1 mot
Initials — 2 mots
Avec avatarUrl
Nom long (edge case)
Usage programmatique 
<!-- Attributs data-* (auto-init au DOMContentLoaded) -->
<div class="user-menu"
     data-display-name="Mike"
     data-email="mike@msyx.fr"
     data-avatar-url=""
     data-authentik-user-url="https://auth.msyx.fr/if/user/"
     data-logout-url="/auth/logout"></div>

<!-- Ou init manuelle avec options JS -->
<div class="user-menu" id="my-user-menu"></div>
<script>
initUserMenu(document.getElementById('my-user-menu'), {
    displayName: 'Mike',
    email: 'mike@msyx.fr',
    avatarUrl: '',
    authentikUserUrl: 'https://auth.msyx.fr/if/user/',
    logoutUrl: '/auth/logout'
});
</script>

La déconnexion soumet un formulaire POST vers logoutUrl — compatible @msyx-dev/auth-node v0.3.0. Keyboard : Flèches haut/bas dans le menu, Escape pour fermer, Tab pour quitter.

Intégration M3 statique (Provider Proxy + Caddy)

Pour un site statique pur (HTML/CSS/JS sans runtime — pattern deploy_mode=static-proxy), <UserMenu> ne peut pas lire la session via un middleware Node. La session est exposée par l'outpost Authentik via des headers HTTP (X-Authentik-*), réémis par Caddy sur /me.json. Le bootstrap se fait côté client après le rendu du header, via l'event msyx:header:ready (pattern event-driven validé sur design-system pour éviter la race condition slot DOM vs script init — cf. pattern-m3-static-proxy §6).

1. Slot DOM placeholder (à insérer dans le header) :

<div class="user-menu" id="ds-user-menu"></div>

2. Bootstrap event-driven (à inclure dans chaque page consommatrice) :

<script>
// Attendre le rendu du header DS avant de fetch + init UserMenu
document.addEventListener('msyx:header:ready', function () {
    fetch('/me.json', { credentials: 'same-origin' })
        .then(function (r) { return r.ok ? r.json() : null; })
        .catch(function () { return null; })
        .then(function (me) {
            var slot = document.getElementById('ds-user-menu');
            if (!slot || !me) return;
            slot.dataset.displayName = me.name || me.username || '';
            slot.dataset.email = me.email || '';
            slot.dataset.authentikUserUrl = 'https://auth.msyx.fr/if/user/';
            slot.dataset.logoutUrl = '/auth/logout';
            if (typeof initUserMenu === 'function') initUserMenu(slot);
        });
});
</script>

3. Côté Caddy (Caddyfile.container) — handles requis :

# /me.json — relaye les headers Authentik injectés par l'outpost edge
handle /me.json {
    header Content-Type "application/json"
    respond `{"name":"{http.request.header.X-Authentik-Name}","email":"{http.request.header.X-Authentik-Email}","username":"{http.request.header.X-Authentik-Username}","groups":"{http.request.header.X-Authentik-Groups}","uid":"{http.request.header.X-Authentik-Uid}"}` 200
}

# /auth/logout — redirection 303 vers le sign_out natif de l'outpost
handle /auth/logout {
    redir * /outpost.goauthentik.io/sign_out 303
}

Référence complète : ~/.claude/projects/-home-deployer/memory/pattern-m3-static-proxy.md — détaille l'archi Authentik Provider Proxy forward_single, les 5 handles canoniques du Caddyfile.container, et les 3 pièges critiques résolus (X-Forwarded-Proto, @public bypass, handle outpost dédié).
